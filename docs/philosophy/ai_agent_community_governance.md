# AI Agentを中心とする自治会運営構想

**作成**: 2026-02-20  
**位置づけ**: CI構想の実装論。哲学的基盤（民主主義の衛生）と接続する。

---

## 1. 発端：GitHub Issues という気づき

住民の要望管理に GitHub の Issues が使えるのではないか、という着想から始まった。

しかしこれは「要望管理ツールの導入」という話ではない。**AI Agent を中心に据えた自治会運営**という、より根本的な構想の入口だった。

---

## 2. このリポジトリ自体がすでに実証している

`COPILOT-INIT.md` + `PROJECT_STATUS.md` という構造は、AI Agent を中心とした運営の**原型がすでに動いている**状態だ。

- 年報の執筆を AI Agent が支援している
- 制度記憶（課題の経緯・議論の背景）が AI との対話ログとして蓄積されている
- 次のセッションでも文脈が引き継がれる

「T氏が AI を使って仕事をしている」のではなく、**「AI Agent が組織の記憶を保持し、意思決定を支援し、文書を生成している」**のだ。  
GitHub Issues の構想は、この閉じたループを「住民 → Issue → AI Agent → 役員会」という開かれたループに拡張する。

---

## 3. 解決される構造的問題

このプロジェクトで繰り返し確認された問題の核心：

> **役員が変わるたびに、組織の知識がリセットされる**

具体的な例：
- 出光石油交差点問題 → 角倉会長が数年間、個人の記憶の中だけで保持していた
- ガードレール撤去 → 議論の経緯が誰にも引き継がれていなかった
- Nさん（第6班）の訴え → 役員会議事録に記載されなかった

AI Agent が組織の中心にいることで：

| 従来 | AI Agent 中心 |
|------|--------------|
| 引継ぎ書を渡す → 受け取った人が読んで理解 | 「AI に聞けばわかる」 |
| 記憶は個人に依存 | 記憶はリポジトリが保持 |
| 役員の交代 = 知識のリセット | 役員の交代 = 対話の仕方を教えるだけ |
| 負荷が特定の人に集中 | 負荷が AI Agent に分散 |

これは**役員のなり手不足問題への直接的な構造的回答**でもある。

## 4. 役員の役割が変わる

現在の役員：「すべてを抱える人」  
AI Agent 中心の役員：「AI Agent が準備した判断材料を最終確認する人」

これは年報の課題2（三役選出の安定性）で検討した心理的負荷軽減の、究極の形だ。  
役職の心理的重圧は「何でもしなければならない」という感覚から来る。  
AI Agent が情報整理・文書生成・経緯説明を担えば、役員は判断だけに集中できる。

## 5. 費用ゼロで動く

この構想の実現可能性を高める決定的な条件：**自治会の予算を使わない**。

| インフラ | 費用 |
|----------|------|
| GitHub リポジトリ（public） | 無料 |
| GitHub Issues | 無料 |
| GitHub Pages（静的サイト） | 無料 |
| AI Agent（Copilot） | T氏の個人契約。自治会費用ではない |

比較対象となる商用ツール（kintone、Notionチーム版等）は月数千〜数万円。  
「費用がかかるから導入できない」という反論が成立しない。

### コストの非対称性

金銭的コストはゼロだが、学習コストは存在する。ただし非対称に配分できる：

- **運営側**（T氏のような人が1人）：GitHub と AI Agent を扱える必要がある
- **住民側**：URL を開いて読めれば十分（アカウント不要）
- **将来の役員**：「AI に聞く」だけでよくなる

コストを1点に集中させることで、全体に分散していたコストを消している。

### 5a. 合法性の検証

「費用ゼロ」は合法か。以下の各層で確認できる。

**利用規約上の適法性**：  
GitHub の利用規約は public リポジトリの無料利用を明示的に認めており、非営利・コミュニティ活動での使用に制限はない。

**自治会が直接契約していないことの意味**：  
自治会は GitHub とも Microsoft とも契約しておらず、費用を負担していない。  
T氏が個人として使用しているサービスを、自治会活動に「活用」しているという構造だ。  
これは「T氏が個人の車で回覧板を届ける」のと本質的に同じであり、問題はない。  
AI Agent（Copilot）についても同様——T氏の個人契約であり、自治会費は一切使われていない。

**個人情報保護の設計**：  
GitHub Issues は public であるため、記録する内容には注意が必要だ。  
設計原則として：
- Issues には個人名・住所・連絡先を記載しない
- 「第6班から要望」のように班単位にとどめる
- センシティブな内容（隣人トラブル等）は Issues に載せない、または非公開チャネルで対応

この設計を守る限り、個人情報保護法上の問題は生じない。

**非公式サイトとしての立場の活用**：  
このサイトはそもそも「有志による非公式のページ」だ。公式組織の制約を受けないことが、  
逆にこの構想を実行しやすくしている。公式の自治会が GitHub を「導入する」のではなく、  
有志が GitHub を「使って支援している」という形は、制度的にもシンプルだ。

### 5b. 思想的必然性——オープンソースの精神的系譜

費用ゼロが可能な理由は、偶然でも慈善でもない。**思想的必然性**がある。

1980年代、Richard Stallman は「ソフトウェアは自由に共有されるべきだ」と主張し、  
GNU プロジェクトを立ち上げた（Free Software = 「自由なソフトウェア」、無料の意味ではない）。  
1991年、Linus Torvalds が Linux カーネルを公開した。  
2001年、Jimmy Wales が Wikipedia を立ち上げた。  
そして Git 自体がオープンソースプロジェクトとして生まれ、それを基盤に GitHub が作られた。

この流れの本質：**「知識・ツールを公共財として開放することで、全体の知性が上がる」**。

GitHub が public リポジトリを無料にしているのは、このオープンソースエコシステムへの投資だ。  
Microsoft が GitHub を買収してもこの方針を維持しているのは、  
オープンソースコミュニティなしには商業エコシステムも成立しないという認識があるからだ。

神南大自治会がこのインフラに乗ることは：
- 先人たちの「知識を公共財に」という思想の**受益者**になること
- 自治会の活動記録・知見を公開することで、他のコミュニティへの**還元者**にもなること

費用ゼロは「タダ乗り」ではなく、**公共財の循環に参加すること**だ。  
これは CI 構想——個人の知識 → AI 学習 → 他者への還元——と同じ構造を持っている。

### Stallman vs Gates：どちらが「動かしているか」

この問いに正直に向き合う。

**Gates の論理**：ソフトウェアは知的財産。報酬がなければ誰も開発しない。商業的インセンティブが技術を前進させる。

**Stallman の論理**：ソフトウェアは知識。知識は人類の共有財産であるべき。蓄積された知識の開放が全体の知性を高める。

インターネットの**基盤インフラ**を見ると：

- OS: Linux（オープン）
- Web サーバー: Apache / Nginx（オープン）
- プロトコル: HTTP, TCP/IP（オープン）
- バージョン管理: Git（オープン）
- AI モデルの基礎研究: 論文として公開（オープン）

**すべてが乗っかっている基盤が Stallman の側にある。**

Gates の後継者（Microsoft / Apple / Google）は今や、この基盤の**上に商業的価値を乗せる**モデルになっている。Microsoft が Linux を敵視した1990年代から、GitHub を買収して OSS を支援する現在への転換は、Stallman の論理が基盤として勝利したことを物語っている。

ただし、素直でない部分もある。ChatGPT / Copilot のような AI は、莫大な商業投資なしには生まれなかった。Stallman が批判する「商業的 AI」が、Stallman 的な活動（このリポジトリの運営）を可能にしているというアイロニーがある。

**統合的な見方**：  
「Stallman が基盤を動かし、Gates がその上の層を動かしている」。  
そして歴史的傾向として、**上の層は時間をかけて基盤に沈降していく**（商業価値は失われ、公共財になる）。

結論として：**Stallman がいないことには始まらない**。  
商業層はその上に乗っているが、基盤なしには成立しない。  
このリポジトリが費用ゼロで動けるのは、まず Stallman のおかげだ。

## 6. 透明性と民主主義の衛生

GitHub Issues は public リポジトリであれば誰でも閲覧できる。

住民が立てた要望、役員会での対応、その結果——すべてが時系列で公開される。  
これは「知らないところで決まる」という不透明な意思決定への**構造的防衛**だ。

[structural_prevention_of_dictatorship.md](structural_prevention_of_dictatorship.md) で論じた「民主主義の衛生」と直結する：  
民主主義 = 独裁者がいない状態ではなく、**構造的防衛が働いている状態**。  
AI Agent による透明な記録はその免疫システムの一部となる。

## 8. CI構想との統合

個人の知識 → AI 学習 → 他者への還元、という CI 構想のループが、  
自治会運営の文脈で具体的な形を持つ：

```
住民の要望・経験
       ↓
  GitHub Issues（記録・公開）
       ↓
  AI Agent（分類・要約・優先順位付け）
       ↓
  役員会（判断・決定）
       ↓
  結果の記録（Issues クローズ・コメント）
       ↓
  住民が確認 → 次の要望へ
       ↓
  （ループ）
```

個人の記憶に依存していた組織が、**集団の知性として機能し始める**。

## 10. なぜ「先行」が必要か——魅力の喪失と回復

### 自治会の歴史的な位置の変化

かつて、自治会は住民より「先を行く」存在だった。

第三層（情報・交流・娯楽・互助）は、個人では手に入りにくかった。集会所での映画会、回覧板による情報共有、冠婚葬祭の助け合い——これらは自治会でなければ得られないものだった。だから加入することに意味があった。自治会は住民の需要の「前」にいた。

今は逆だ。情報はスマホで、娯楽はサブスクで、交流はSNSで得られる。自治会が提供するものを、個人がすでに持っている。**自治会が住民の「後ろ」に回ってしまった。**

### 後退した組織の必然的な帰結

魅力を失った組織が取る発想は2種類ある：

**守勢の発想**（悪循環）：魅力がない → 懲罰で引き留める → さらに魅力を失う  
→ ゴミ集積所の高額使用料（¥12,000）はその象徴だ。「来たければどうぞ」ではなく「逃げるなら罰を与える」。組織として守勢に入っている証拠だ。

**先行の発想**（好循環）：時代より前を行く → 魅力を創る → 引き寄せる

### 「先行」はすべての層に効く

三層モデルのA・B・Cタイプすべてに、先行する自治会は異なる形で魅力を持つ：

- **Aタイプ**：透明で効率的な運営 → 会費が適切に使われているという**信頼**
- **Bタイプ**：引継ぎが途切れない安定した組織 → 第2層への期待に応える**安心感**
- **Cタイプ**：時代の前を行くコミュニティ → 参加することへの**誇りと面白さ**
- **役員のなり手**：能力があり将来を考えられる人ほど、後退した組織には関わりたくない。「この自治会は面白い」と思える人が役員になりたがる。**なり手不足の根本的な解決はここにある。**

### AI Agent 構想の位置づけ

AI Agentによる組織的記憶・透明な要望管理・引継ぎの自動化は、個人が自力では絶対に持てないものだ。費用もかからない。自治会がこれを先に持つことは、**昔の自治会が持っていた「先行性」を、AI時代の形で取り戻すこと**だ。

懲罰で引き留めようとする発想と、先行することで引き寄せる発想は、まったく逆の方向を向いている。

### 伝え方について

「AI Agent」「GitHub」という言葉を使わなくても、この構想は伝わる：

> 「役員が変わっても記録が積み重なり、引き継がれていく仕組みを、費用をかけずに整え始めています」

技術の名前より「役員が変わっても大丈夫」という事実が届く。そして「この年報も、その仕組みの一部です」という一文が、理解できる人には深く届くBGMになる。

### 観音台の地域構造

- 観音台は第一自治会（約220世帯）と第二自治会（約114世帯）の2つに分かれている
- 両自治会は集会所を共同利用し、毎年合同役員会を開く関係
- 第一自治会の山岡会長は来年度も顧問として役員会に参加予定

### GitHub Organization 構造（案A）

```
kannondai org/
  kannondai-community/   # 観音台共通（集会所予約・地域情報・環境）
  jichikai-2/            # 第二自治会の内部活動（年報・議事録・要望管理）
  jichikai-1/            # 第一自治会（山岡さんが参加する段階で作成）
```

- 共通情報と各自治会の内部情報を分離できる
- 将来、山岡さんを `jichikai-1` の管理者として招待することが自然な流れになる

### 移行スケジュール

| タイミング | 作業 |
|-----------|------|
| 2026-02-20 | org `kannondai` 確保済み ✅ |
| 3/28 総会後 | `kannondai-community` リポジトリを org に移管 |
| 来年度初め | `jichikai-2` リポジトリを作成、内部文書を分離 |
| 山岡さんと合意後 | `jichikai-1` リポジトリを作成、招待 |

**基本方針**：無理のないスピードで。年報・総会への集中を優先し、移管は総会後。
